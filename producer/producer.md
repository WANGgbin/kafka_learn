# 发送客户端架构
producer 通过异步的方法发送消息. 为什么要采用异步的发送方式呢?

我们都知道 producer 高吞吐量的一个很重要的原因就是消息的批量发送, 如果采用同步发送的方式,那么什么时候才应该真正发送当前批次的消息呢? 我们可能想到的是,每次产生一条消息的时候,可以判断当前消息的数量是否达到一定值,是的话就发送当前批次数量. 可是如果产生部分消息后,不再产生消息,当前批次的消息是不是就发送不出去了呢? 可以设置超时时间? 同步方式又无法检测超时时间.

因此, producer 采用异步的方式发送消息. producer 里面有两个线程,一个主线程,一个发送线程. 主线程负责把消息写到缓冲区中, sender 线程负责发送数据到 broker.

缓存区大小怎么设置呢? 通过 `buffer.memory` 参数设置,单位 Byte, 默认值为 32MB.

如果缓冲区满了,主线程会阻塞吗? 可以通过参数 `max.block.ms` 来设置阻塞时间,默认是 60s.

可以通过参数 `batch.size` 来设置 ProduceBatch 的大小.

通过参数 `linger.ms` 来设置消息等待最大等待时间, 默认值是 0. 调大该值,可以一定程度的提高吞吐量,但同时会导致消息的延迟.

producer 发送消息的时候,会对消息进行分区. 在缓冲区中的消息是根据 partition 存放的, 结构类似为 map<partition, list<produceBatch>>. 因为 sender 负责把消息发送给 broker, 所以 sender 会把消息转化为 map<broker, list<request>>. 这里的 broker 为分区 leader 副本所在的 broker.

这里有个重要的参数 `max.in.flight.requests.per.connection`, 表示每一条跟 broker 的连接可以缓存多少还未收到响应的 request. 这种机制进一步提高了发送消息的吞吐量. 可以在接受到消息的响应前, 进一步发送多个请求, 让发送端和接受端都忙起来,避免等待请求/响应上的时间浪费. 但是,另一方面,当有很多消息都没有收到响应的时候,说明接受端有点 '忙', 此时发送消息会阻塞.

综上, 站在发送客户端的角度, 消息的批量异步发送 和 同一条连接可以发送多个请求 保证了消息发送的高吞吐.

# 其他重要的生产者参数
- acks

表示什么样的条件下,发送者认为成功发送了一条消息.
可取的值有 -1, all, 0, n

-1, all: 表示分区的所有 ISR 中的 broker 都收到消息后, 才认为消息发送成功.
0: 不关心 broker 有没有收到消息, 只负责发送.
n: 表示分区所在的 ISR 中的 n 个 broker 收到消息后, 才认为i消息发送成功.

设置为 0 的情况下,有更高的性能, 但同时可靠性最差,消息可能丢失.
设置为 -1/all, 可靠性最好,消息不会丢失,但是性能最差

acks 的默认值为 1, 即只要分区的 leader 只要收到消息即可.

- retries

该参数用来配置生产者发送消息重试的次数.当发送消息失败的时候,客户端会进行重试. 该参数的默认值为0.

- retry.backoff.ms

该参数用来设置两次重试之间的时间间隔,避免无效的重试同时避免重试流量太大, 该参数的默认值为 100ms.

- request.timeout.ms

该参数用来配置 producer 等待响应的最大等待时长, 默认值为 30000 ms.

# producer 是如何分区的
默认的分区策略是, 如果消息的 key 不为空,则会对 key 进行哈希. 如果 key 为空, 则采用轮询的方式进行分区.

我们可以通过实现分区接口,同时在 producer 的配置中指定分区器来实现自定义的分区.

# 如何保证消息生产的有序
kafka 只保证同一分区消息的有序性. 因此, 如果要让某些消息有序,可以让这些消息发送到同一个分区之中.
实际上,即使同一个分区也有可能导致消息的乱序. 这是由 `retries` 和 `max.in.flight.requests.per.connection`导致的. 如果 max.in.flight.requests.per.connection 的值大于1 同时 retries 的值大于1, 如果前面的消息发送失败了,然后重试就会导致消息的乱序. 通常要保证有序,我们需要将 max.in.flight.requests.per.connection 的值设置为1,但是这样会导致发送消息的吞吐量降低.
